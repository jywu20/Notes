
HOL这个名字有一定误导性，它其实是应该要理解为一种高阶逻辑，而不是the higher order logic。传统数理逻辑语境下（从语义角度出发定义的标准语义的）高阶逻辑是没法有效公理化的，所以也无所谓the higher order logic。

一个一般的（语形意义上的，作为推理系统的）高阶逻辑为了避免谓词版本的罗素悖论，需要有类似类型的机制。否则可以定义谓词P(x)为“x让x(x)不成立”，然后我们可以问P(P)成立不成立，就悖论了。最直接的解决这个问题的办法就是让x(x)这种式子写不出来，于是你就得引入类型。但是这里的类型当然可以是罗素和怀特海那种类型，保证式子well formed以后就不再出现。但是HOL里面的类型真的就是编程语言里面那个类型，A->B，这种东西。谓词真的就是从某个类型到bool的函数，高阶谓词是(A -> bool) -> bool，这样子。

我们今天所谓的HOL最早应该是Church定义的，他称为simple type theory。注意这个不是simply typed lambda calculus；HOL里面有命题，STLC里面没有。这里要注意一件事，就是HOL里面的逻辑推理不是靠Curry-Howard同构的：我们有bool类型的term，即命题，可以理解成经典CIC+proof irrelevance里面的Prop的term，但是标准HOL里面的命题本身不是类型，所以也没有proof term。HOL里面的命题是基于等式的。所以它不需要全称量词引入规则或者消去规则，因为forall x:A. P(x) 可以直接定义成lambda x:A. P(x) = lambda x:A. True.

这么讲，一般的HOL都是经典的。其实也有构造主义的HOL，Isabelle/Pure就算，但那个太弱了没法不加公理直接用。害其实Lean能火，我看和它的经典逻辑就是给人用的有关系。主流数学界就是只想听“是”和“不是”。

所以HOL很朴实，基本上就是习惯了普通（i.e. no dependent type）编程语言的人设想的“同时能做计算和证明的环境”该是的样子。罗素和怀特海的类型论里面没有explicit的计算，而HOL里面有。（我们是不是可以把在HOL做证明当成是在手动解决某些等式的不可机器判定？）

没有proof term会导致有些东西不太方便。如果你想写“定理1：某某某陈述，证明：……”，在Lean里面你是可以直接复用引入变量的语法：可以写 theorem thm1 : xxx := ... 因为这里thm1是一个proof term的名字，而由于proof irrelevance，对数学家来说thm1就可以看成指代了一个已经证明的命题的记号。在HOL里面证明是证明变量定义是定义，导致一些问题诸如你没法把数学对象及其性质打包成一个term传来传去，所以很多东西都要同时做两套语法。比如说“设有G，G、其上的乘法运算满足如何如何的性质，从而形成群结构”，在Lean里面乘法运算、乘法运算的性质可以统一作为type class fields引入，在Isabelle/HOL里面你就得靠Isabelle提供的locale做这个表述，然后写fixes ... assumes ...。当然对数学家来说这个问题可能其实不大因为命题和对象的区分也挺直观的。（可能HOL总是在一个LCF风格的prover里面实现和这个也有关系，因为你肯定需要一些机制来把证明好的定理传来传去以及proof engineering）

我们今天用的HOL普遍还引入了两个扩展，一个是无穷公理（如见Isabelle/HOL的theory Nat）,一个是选择公理（一般会做成希尔伯特选择算符）。这个版本的HOL和bounded Zermelo + Choice是可以互相诠释的，因此equiconsistent。（在类型论中编码集合论是不容易的，从HOL到bounded Zermelo的做法是将一个term a，“x=a”这个谓词，“x=前述谓词”这些东西看成等价的）这样只能编码bounded Zermelo，没有替换公理，因为替换公理里面的那个替换函数可以不是集合论函数，所以某种意义上是允许我们站在世界外面看到世界的一些结构。这个东西在类型论里面肯定不好做，但如果你有Type 0, Type 1这样一系列宇宙就能做到，所以Lean（CIC+Choice+proof irrelevance）和ZFC+countable inaccessible cardinals等价，但没有类型论宇宙，那证明论强度就一下子下去了。

不过其实这样也没什么不好，因为按照范畴论专家Mac Lane的说法，bounded Zermelo + Choice真的就正正好囊括了所有具体的数学。所以相应的HOL也就正正好囊括了所有具体的数学。你不玩序数，不要求字面地形式化Set范畴，那HOL就足够好用。而且HOL这么弱也有很大好处：自动化会更好做。

所以我有一种感觉，就是HOL这个数学基础给人一种活化石的感觉：它稳定下来以后就没什么人去twist它，让它变强或者变得不更弱也不更强。另一方面dependent type那边热衷于穷折腾的人是很多的。HOL和罗素那个系统是等价的，但有了无穷公理和选择公理肯定不能叫逻辑主义了（罗素的PM也受到同样的攻击），但搞neologicism的人和玩HOL的好像也没啥交集。

HOL圈子主要的研究都是关于怎样给它加保守扩张。typedef就是一个保守扩展，forall x:B, ...（其中B是对类型A定义子集以后typedef得到的）可以规约到forall x:A (type guard -> ...)这样的形式上面。归纳类型是通过无穷公理+typedef定义的。当然有些归纳定义是要强过无穷公理的，所以bounded Zermelo里面支持多少归纳定义其实是有讲究的，HOL圈子里面有一个叫BNF的条件（不是语法那个）保证归纳定义能做。Isabelle里面还有模板、有typeclass（每加一个扩展都得发一篇文章证明保守性），现在还有人在做给HOL加dependent type。我忘了是Lawrence Paulson还是谁了还锐评说DT那些人整天改基础，不像HOL，靠一个很简单的基础最后什么功能都齐全了。

但不论功能齐全不齐全吧，HOL现在好像没啥人玩了，很可惜。数学界好像都没啥人知道有这个东西。而且HOL的工具也看着属实原始。HOL4现在你去搜教程还有很多教你如何在ML里面写HOL4的，对编程语言不熟的人看到这种东西直接懵逼了。HOL Light那个记号更是闪瞎狗眼，80年代风过重（全大写，没有unicode数学符号）。Isabelle/HOL至少看着像数学但Isabelle/VSCode现在还是比不上jEdit版的IDE，但现在谁用jEdit啊。而且我也搞不懂他们为什么要停用那个VSCode插件而要自己打包一套。隔壁Lean就一直是插件。

Isabelle对刚接触的人来说还有一个叠屋架床的问题，冒号有两种箭头有三种，Isar关键字多如牛毛，虽然写完好看懂是真的。（有懂哥懂姐知道为何他们要用=>表示函数箭头吗？）

不过据说HOL用在形式验证上面很多。可能也好理解，因为形式验证中的很多中间结果真的没有让人手动去证明的价值，自动化很重要，那HOL自然有优势。

当然这些都是很皮毛的东西，具体实际使用过程中的体验、经验我是不懂半点了，只能有待有缘人回答——如果有的话。

其实我感觉HOL这个思路说不定可以用在教学上面。60年代的新数学运动失败，因为你毕竟不可能让初学者认识到为何要把东西编码到集合上，那自然大家也看不出集合论有啥用。但HOL这个从ind和lambda函数生万物的做法其实“直观”的，以此为骨干向学生教授比较严格的数学，说不定有效。当然我们这个ai时代可能也没人关心数学教学了。这么讲，或许可以探讨一下ai对prover的自动化的有效程度和prover的基础理论的关系。
